СЛАЙД 2 — ПРОБЛЕМА
════════════════════════════════════════════════════════════

[ТЕКСТ ДЛЯ СДАЧИ] (~45 сек)
Три сервера с разной производительностью: 100, 80 и 60 запросов в
секунду. При равном распределении каждый получает по 50 req/s.
Быстрый сервер справляется легко, а медленный работает на пределе:
W₂ = 1/(60−50) = 100 мс. Среднее по системе — 51 миллисекунда.
Задача — найти такие доли трафика xᵢ, чтобы среднее время отклика
Wavg было минимальным.

---
[ДОПОЛНИТЕЛЬНО]
Почему медленный сервер "тянет вниз всю систему:
  Wavg = x₀·W₀ + x₁·W₁ + x₂·W₂
  При равных долях: (20 + 33 + 100) / 3 = 51 мс
  Сервер 2 вносит 100 мс при весе 1/3 — это и портит среднее.

Вопрос: "А почему не убрать медленный сервер?"
  Тогда суммарная пропускная способность падает с 240 до 180 req/s.
  Задача — использовать все серверы оптимально, а не выбрасывать мощности.

СЛАЙД 3 — МОДЕЛЬ
════════════════════════════════════════════════════════════

[ТЕКСТ ДЛЯ СДАЧИ] (~60 сек)
Каждый сервер моделируется как очередь M/M/1. Первая M —
пуассоновский входящий поток, вторая M — экспоненциальное время
обслуживания. Среднее время пребывания запроса: W = 1/(μ−λ).
Загрузка ρ = λ/μ должна быть меньше единицы — это условие
устойчивости. Зависимость нелинейная: при ρ близком к 1 время
отклика стремится к бесконечности.

---
[ДОПОЛНИТЕЛЬНО]
Что значит M/M/1:
  Первая M — Markovian (Пуассоновский) поток. Интервалы между заявками
             случайны, без памяти о прошлом.
  Вторая M — экспоненциальное время обслуживания, тоже без памяти.
  1         — один канал обслуживания.

Почему зависимость нелинейная:
  W = 1/(μ−λ). При ρ=0.5: W = 1/(μ·0.5). При ρ=0.9: W = 1/(μ·0.1) —
  в 9 раз хуже, хотя нагрузка выросла лишь вдвое.

Вопрос: "Почему именно M/M/1, а не другая модель?"
  M/M/1 — аналитически решаемая модель с закрытой формулой.
  Для прототипа достаточно: M/M/1 даёт правильное направление
  оптимизации. Мы подтвердили это SimPy-симуляцией (расхождение < 2%).

СЛАЙД 4 — ТРИ МЕТОДА
════════════════════════════════════════════════════════════

[ТЕКСТ ДЛЯ СДАЧИ] (~90 сек)
Мы реализовали три постановки задачи оптимизации. Общее условие:
найти доли xᵢ, сумма которых равна единице, xᵢ ≥ 0.

LP balanced — минимизирует максимальную загрузку среди серверов:
min max(ρᵢ). Решается через PuLP, целевая функция и ограничения
линейные. Результат — серверы загружены одинаково по ρ.

LP proxy — линейное приближение времени отклика. W нелинейна,
поэтому берём суррогат: min Σ λxᵢ/μᵢ². Тоже решается через PuLP.
Работает хорошо при низкой нагрузке, при высокой — приближение
ломается.

NLP SLSQP — минимизирует настоящий Wavg = Σ xᵢ/(μᵢ−λxᵢ) напрямую
через SciPy. Функция выпуклая, поэтому гарантированно находим
глобальный минимум.

---
[ДОПОЛНИТЕЛЬНО]
Почему LP proxy использует λxᵢ/μᵢ²:
  Тейлоровское разложение при малом ρ:
  W_i = 1/(μᵢ−λᵢ) ≈ 1/μᵢ + λxᵢ/μᵢ²
  Константу 1/μᵢ отбрасываем — она не зависит от xᵢ.
  Остаётся линейная функция, доступная LP.

Вопрос: "Зачем три метода, если NLP всегда лучше?"
  LP решается мгновенно и детерминированно — без итераций.
  NLP итерационный, в граничных случаях может потребовать больше
  времени. LP balanced полезен когда важна "честность" — равномерная
  загрузка, а не только минимум времени.

Вопрос: "Что такое выпуклая задача?"
  Единственный минимум — нет "ям" куда алгоритм может провалиться.
  Любой локальный минимум является глобальным. Сумма выпуклых функций
  выпукла, а xᵢ/(μᵢ−λxᵢ) выпукла при xᵢ > 0.

Вопрос: "Что такое SLSQP?"
  Sequential Least Squares Programming — итерационный метод. На каждом
  шаге строит квадратичное приближение задачи и решает его. Обычно
  10–30 итераций до сходимости.


════════════════════════════════════════════════════════════
СЛАЙД 5 — РЕЗУЛЬТАТЫ
════════════════════════════════════════════════════════════

[ТЕКСТ ДЛЯ СДАЧИ] (~60 сек)
λ = 150 req/s, серверы μ = 100, 80 и 60.

Baseline — равные доли: Wavg = 51.1 мс.

NLP даёт 32.8 мс — лучший результат, снижение на 36% относительно
baseline, как видно на графике.

LP balanced даёт 33.3 мс — практически то же самое, хотя он вообще
не оптимизирует W напрямую, а выравнивает загрузку.

LP proxy — 80 мс, хуже baseline. Это ожидаемо: при высокой нагрузке
ρ ≈ 0.9, линейное приближение некорректно. Метод отправляет весь
трафик только на два быстрых сервера — и загоняет их в перегрузку.

---
[ДОПОЛНИТЕЛЬНО]
Почему LP proxy = 80 мс (хуже 51 мс baseline):
  Смотрим на следующий слайд — там виден ответ:
  LP proxy ставит x₂ = 0 (нулевой трафик на медленный сервер).
  Всё уходит на сервер 0 (x₀=0.6) и сервер 1 (x₁=0.4).
  λ₀ = 90 req/s при μ₀=100 → ρ₀=0.9 → W₀ = 1/(100−90) = 100 мс
  λ₁ = 60 req/s при μ₁=80  → ρ₁=0.75 → W₁ = 1/(80−60)  = 50 мс
  Wavg = 0.6·100 + 0.4·50 = 80 мс — хуже baseline.
  Линейный суррогат не "видит" взрывного роста W при ρ→1.

Как считали улучшение:
  (51.1 − 32.8) / 51.1 × 100% = 35.9% ≈ 36%.

СЛАЙД 6 — ТРАФИК
════════════════════════════════════════════════════════════

[ТЕКСТ ДЛЯ СДАЧИ] (~40 сек)
На графике — доли трафика xᵢ по каждому методу.

LP balanced распределяет пропорционально μ: больше быстрым,
меньше медленным — но равномерно по загрузке.

NLP делает похожее, но чуть агрессивнее отдаёт быстрым серверам.

LP proxy — крайний случай: отправляет 60% на сервер 0, 40% на
сервер 1 и ноль на медленный сервер 2. Именно это и объясняет
результат 80 мс с предыдущего слайда.

---
[ДОПОЛНИТЕЛЬНО]
Конкретные числа (из кода):
  LP balanced: x₀=0.417, x₁=0.333, x₂=0.250  (пропорционально μ)
  NLP:         x₀=0.442, x₁=0.332, x₂=0.226
  LP proxy:    x₀=0.600, x₁=0.400, x₂=0.000  (←  нулевой трафик на S2)
  Равные доли: x₀=x₁=x₂=0.333

Вопрос: "Почему NLP не отдаёт ещё больше трафика быстрому серверу?"
  Ограничение ρ_max = 0.9. При x₀ > 0.6 загрузка сервера 0 превысит
  0.9 — это нарушает ограничение устойчивости.

Вопрос: "LP balanced и NLP дают почти одинаковый W (33.3 vs 32.8) —
зачем тогда NLP?"
  На этих параметрах разница мала. При более неравномерных μ или
  другом λ разрыв будет больше. NLP — математически точный минимум.


════════════════════════════════════════════════════════════
СЛАЙД 7 — АРХИТЕКТУРА
════════════════════════════════════════════════════════════

[ТЕКСТ ДЛЯ СДАЧИ] (~45 сек)
Система реализована на Python. Ядро — три модуля: queue_math.py
с формулами M/M/1, lp_optimizer.py с тремя методами оптимизации
и simulation.py с дискретно-событийной симуляцией через SimPy.

Поверх — FastAPI с эндпоинтами /balance, /compare и /health.
Балансировщик принимает параметры, считает оптимальное
распределение и возвращает доли в реальном времени. Всё упаковано
в Docker и запускается одной командой.

Валидация SimPy подтвердила аналитические формулы: расхождение
менее 2%.

---
[ДОПОЛНИТЕЛЬНО]
Эндпоинты FastAPI:
  POST /balance  — принимает λ, список μ, метод → возвращает xᵢ и метрики
  GET  /compare  — сравнивает все три метода сразу
  GET  /health   — проверка живости сервиса
  GET  /         — веб-дашборд

Вопрос: "Зачем Docker?"
  Контейнер изолирует зависимости — сервис работает на любой машине
  без установки Python, PuLP, SciPy. docker-compose up — и готово.

Вопрос: "Почему расхождение < 2%, а не 0%?"
  M/M/1 даёт стационарное решение — верно при бесконечном времени.
  Симуляция конечная (3000 сек модельного времени), поэтому есть
  статистическая погрешность. Это нормально для прототипа.

Вопрос: "Как работает SimPy?"
  Дискретно-событийная симуляция: генерирует случайные заявки,
  ставит их в очередь, обрабатывает — и измеряет фактическое
  время ожидания. Библиотека SimPy на Python.

СЛАЙД 8 — ВЫВОДЫ
════════════════════════════════════════════════════════════

[ТЕКСТ ДЛЯ СДАЧИ] (~30 сек)
Итог: NLP SLSQP даёт наилучший результат — 32.8 мс против
51 мс при равномерном распределении, снижение на 36%.
LP balanced — хорошая детерминированная альтернатива с результатом
33.3 мс. LP proxy работает только при низкой нагрузке.
Модель M/M/1 подтверждена симуляцией. Система готова к
развёртыванию через FastAPI + Docker.

---
[ДОПОЛНИТЕЛЬНО]
Вопрос: "Где это применяется в реальной жизни?"
  Nginx, HAProxy, AWS ALB — все балансировщики решают похожую задачу.
  Взвешенный round-robin (трафик пропорционально μ) лучше равномерного
  именно потому, что M/M/1 нелинейна. Мы это математически обосновали.

Вопрос: "Какие ограничения у модели?"
  M/M/1 предполагает независимые запросы и экспоненциальное
  обслуживание. В реальности сессии могут быть связаны (M/M/1/K),
  обслуживание может быть детерминированным (M/D/1). Направление
  для развития — более сложные модели очередей.

Вопрос: "Что если NLP не сходится?"
  Функция выпуклая — при корректных данных (ρ_max < 1, Σμ > λ)
  SLSQP всегда сходится. Код возвращает res.success=False и
  API отвечает 422 если что-то пошло не так.

ШПАРГАЛКА — КЛЮЧЕВЫЕ ЧИСЛА
════════════════════════════════════════════════════════════

  λ=150 req/s, μ=[100, 80, 60], ρ_max=0.9

  Равные доли:   Wavg = 51.111 мс  (x = 0.333 / 0.333 / 0.333)
  LP balanced:   Wavg = 33.333 мс  (x = 0.417 / 0.333 / 0.250)
  LP proxy:      Wavg = 80.000 мс  (x = 0.600 / 0.400 / 0.000)  ← хуже baseline
  NLP SLSQP:     Wavg = 32.768 мс  (x = 0.442 / 0.332 / 0.226)  ← лучший

  Улучшение NLP vs baseline: (51.111 − 32.768) / 51.111 = 35.9% ≈ 36%
  Валидация SimPy: расхождение < 2%

  ФОРМУЛЫ:
  W_i = 1 / (μᵢ − λᵢ)     λᵢ = xᵢ · λ
  ρᵢ  = λᵢ / μᵢ           Wavg = Σ xᵢ · Wᵢ
